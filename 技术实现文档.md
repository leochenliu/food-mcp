# McAgent 麦当劳智能订餐调度系统 — 技术实现文档

> 项目名称：`food-mcp`（AI Studio 导出工程 `McAgent 麦当劳智能订餐调度系统 v2.4.0`）
> 文档版本：v1.0 ｜ 对应代码快照：main 分支工作树

---

## 1. 项目概述

本项目是一个**交互式 AI 订餐演示系统**，用于可视化展示一种典型的「大脑与四肢分离（Brain & Limbs）」智能体架构：

- **大脑（Brain）**：以 Markdown 形式存在的「Agent Skills（业务规则书）」，由大语言模型（MiniMax-M3）读取并据此推理、决策、发起工具调用。规则可热重载、可开关，无需重新编译。
- **四肢（Limbs）**：一个模拟运行在 Cloudflare Worker 上的 **TypeScript MCP 服务端**，负责执行原子级、事务型、受保护的底层操作（加购、绑券、锁单、创建订单等），并把核心交易逻辑与 AI 文本输入严格隔离，防止提示词注入篡改资金数据。

系统通过环境变量（天气 / 距离 / 时间 / 积分）模拟真实门店场景，用户可以：
1. 直接点击菜单加购，或
2. 用自然语言向 AI 客服「Mickey」下达指令，

两种路径都会触发同一套 MCP 工具，并在界面右侧的「MCP 服务端事务处理层」实时打印执行日志与返回载荷。

## 5. 后端实现（`server.ts`）

### 5.1 服务启动与双模式

`startServer()` 根据 `NODE_ENV` 决定静态资源来源：

- **开发模式**：调用 `createViteServer({ middlewareMode: true, appType: "spa" })`，把 Vite 中间件挂到 Express，实现 HMR 与按需编译。
- **生产模式**：`express.static("dist")` 托管打包产物，并对 `*` 兜底返回 `index.html`（SPA 路由）。

监听端口 `PORT = 3000`，绑定 `0.0.0.0`。

### 5.2 MiniMax 客户端（OpenAI 兼容 + 安全降级）

```ts
const MINIMAX_BASE_URL = "https://api.minimaxi.com/v1"; // 域名为 api.minimaxi.com，非 minimax.io
const MINIMAX_MODEL = "MiniMax-M3";

async function callMiniMax(payload) {
  const res = await fetch(`${MINIMAX_BASE_URL}/chat/completions`, {
    method: "POST",
    headers: { "Authorization": `Bearer ${process.env.MINIMAX_API_KEY}`, "Content-Type": "application/json" },
    body: JSON.stringify({ model: MINIMAX_MODEL, ...payload })
  });
  if (!res.ok) throw new Error(`MiniMax API error ${res.status}: ${await res.text()}`);
  return res.json();
}
```

直接通过 `fetch` 调用 MiniMax 的 OpenAI 兼容 `POST /v1/chat/completions` 接口，使用 `Authorization: Bearer <MINIMAX_API_KEY>` 认证。若未设置 `MINIMAX_API_KEY`，仅打 warning，不抛错；运行时走 Mock 分支。**API Key 永不进入前端**。

### 5.3 核心定价算法 `recalculateOrder(state, env, combosSkillActive, promoSkillActive)`

统一的价格计算函数（前端 `App.tsx` 内 `recalcTotal` 与之逻辑镜像，保证 UI 与后端一致）：

1. **基础小计**：`subtotal = Σ(item.price × quantity)`。
2. **优惠券折扣**（仅当 `promoSkillActive`）：
   - `MCD50`：满 `$50` 立减 `$10`。
   - `MCD_BREAKFAST_15`：仅早餐类（`BRK_` 前缀）`85` 折。
   - `MCD_BURGER_20`：指定汉堡类 `8` 折。
3. **套餐组合返利**（仅当 `combosSkillActive`）：
   - 早餐组合：麦满分 + 脆薯饼 + 咖啡同时存在 → 每组 `-$2.50`（按最小数量为组数）。
   - 正餐组合：巨无霸/麦辣鸡腿堡 + 薯条 + 可乐同时存在 → 每组 `-$1.50`。
4. **配送费**：`isDelivery && (distanceKm > 2.0 ? 4.50 : 2.50)`，自提为 `$0`。
5. **税费**：`(subtotal - discount - bundleCredit) × 8.5%`。
6. **总价**：`subtotalWithDiscount + deliveryFee + tax`，全程 `toFixed(2)` 防浮点误差。

### 5.4 MCP 工具声明（`FunctionDeclaration`）

共声明 8 个工具（OpenAI 兼容 `tools` 数组，每项为 `{ type: "function", function: { name, description, parameters } }`）：

| 工具名 | 入参 | 业务含义 |
|--------|------|----------|
| `set_delivery_scenario` | `isDelivery` | 切换外卖/自提，更新配送费 |
| `bind_coupon` | `code` | 绑定优惠券（校验码 + 时段 + 满减） |
| `redeem_points_hashbrown` | 无 | 扣 500 积分赠免费脆薯饼 |
| `lock_order` | 无 | 冻结订单价格与库存（防篡改） |
| `create_order` | 无 | 生成 `MCD-XXXXXX` 订单号与取餐码 |
| `add_item_to_cart` | `sku`, `quantity` | 加购（校验库存 + 早餐时段闸门） |
| `remove_item_from_cart` | `sku` | 移除某项 |
| `list_nutrition` | 无 | 汇总购物车营养指标 |

### 5.5 接口 `/api/agent/chat`（POST）

**请求体**：

```json
{
  "messages": [{ "sender": "user"|"agent", "text": "..." }],
  "environment": { "weather", "distanceKm", "timeOfDay", "userPoints", "isDelivery" },
  "orderState": { ... },
  "activeSkills": ["skill_scenario", "skill_promotion", ...]
}
```

**处理流程**：

1. 用 `activeSkillIds` 过滤出激活的 Skills，拼装为 `systemInstruction`（含环境上下文、菜单实时库存、当前购物车、工具使用准则）。
2. **无 Key → Mock 分支**：基于用户最新消息关键词（delivery / coupon / add / points / checkout）执行本地规则引擎，构造 `mcpCalls` 与 `updatedOrder`，直接返回。
3. **有 Key → MiniMax 分支**：
   - 第一次调用 `callMiniMax`，传入 `messages`（system + 历史）+ `tools` + `tool_choice: "auto"`，模型在 `choices[0].message.tool_calls` 返回工具调用。
   - 遍历 `tool_calls`，`JSON.parse` 解析 `function.arguments` 后逐个执行对应事务逻辑（写回 `updatedOrder` / `finalEnvironment`，记录 `toolCalls`）。
   - 调用 `recalculateOrder()` 重算金额。
   - 第二次调用 MiniMax，把工具执行结果回填，生成友好的自然语言回复（含支付 Deep Link 引导）。
4. 返回 `{ text, orderState, mcpCalls, environment }`。

**错误兜底**：异常时返回 `500 { error }`，前端捕获后插入 system 消息提示。

### 5.6 安全与隔离设计

- 订单号 `order_id`、取餐码由**服务端随机生成**，AI 文本无法伪造（见 `EnvironmentControl` 合规警示）。
- `lock_order` 在结算前冻结状态，防止 AI 在用户付款瞬间更改总价。
- 库存校验（`stock < quantity`）、早餐时段闸门（`timeOfDay >= 10:30` 禁止早餐品）在服务端强制拦截，即使前端绕过也无效。

## 6. 前端实现（`src/`）

### 6.1 全局状态（`App.tsx`）

`App` 组件集中管理 6 块状态：

| 状态 | 类型 | 说明 |
|------|------|------|
| `env` | `Environment` | 天气 / 距离 / 时间 / 积分 / 是否配送 |
| `skills` / `activeSkillIds` | `AgentSkill[]` / `string[]` | 规则书数据与激活集合 |
| `orderState` | `OrderState` | 购物车 + 定价账目 + 订单状态机 |
| `logs` | `McpToolCall[]` | MCP 执行日志累积流 |
| `messages` | `ChatMessage[]` | 聊天记录 |
| `isThinking` | `boolean` | AI 思考态标志 |

### 6.2 自动重算 Effect

```ts
useEffect(() => {
  if (!isBreakfast) { /* 切换正餐时段时清除购物车中的早餐品 */ }
  recalcTotal();
}, [env.timeOfDay, env.isDelivery, env.distanceKm, orderState.items, orderState.appliedCoupon, activeSkillIds]);
```

- 时段从早餐切到正餐（≥10:30）时，自动移除购物车早餐项，并写入 `catalog_hour_check` 警示日志。
- 依赖项变化即触发前端侧定价重算，保证 UI 即时反映 Skills 开关、配送方式、优惠券的影响。

### 6.3 前端 MCP 工具代理

UI 直接操作（加购/减购/绑券/兑换/锁单/创建订单/清空）均由对应的 `handleXxx` 函数处理：

- 先 `setOrderState` 更新购物车；
- 同步调用 `triggerMcpLog()` 生成一条 `McpToolCall` 并 push 到 `logs`，模拟「服务端已执行」。
- 订单处于 `locked` / `paid` 时，加购与减购被直接拦截（不可变）。

### 6.4 与后端对话 `handleSendMessage`

```ts
const response = await fetch("/api/agent/chat", {
  method: "POST",
  body: JSON.stringify({ messages, environment: env, orderState, activeSkills: activeSkillIds })
});
```

拿到响应后：
- 追加 agent 聊天消息（携带 `mcpCalls` 子卡）；
- 把 `data.mcpCalls` 合并进 `logs`；
- 用 `data.environment` 与 `data.orderState` **覆盖式同步**本地状态，保证前端状态机与后端结算结果完全一致。

## 7. 数据模型（`src/types.ts`）

核心接口关系：

```
MenuItem { sku, name, category, price, stock, nutrition... }
   │ 1:N
   ▼
CartItem { item: MenuItem, quantity }
   │ 聚合
   ▼
OrderState { status: idle|draft|locked|paid, items[], appliedCoupon,
             subtotal, discount, bundleCredit, deliveryFee, tax, total,
             id, pickupCode }

Coupon { code, discountType: fixed|percentage, value, minSpend, categoryLimit }
AgentSkill { id, title, category: scenario|promotion|pairing|payment, content }
Environment { weather, distanceKm, timeOfDay, userPoints, isDelivery }
McpToolCall { id, timestamp, toolName, args, result, status, type: read|transaction, message }
ChatMessage { id, sender, text, timestamp, mcpCalls? }
```

**订单状态机**：`draft` →（lock_order）→ `locked` →（create_order）→ `paid`。`locked`/`paid` 后禁止修改购物车。

## 8. 业务逻辑要点

1. **时段闸门**：`env.timeOfDay < "10:30"` 为早餐时段；后端 `add_item_to_cart` 对早餐品做硬拦截，前端 `MenuGrid` 也仅展示对应时段商品。
2. **优惠券互斥**：每个订单最多一张券；积分兑换奖励可**叠加**优惠券（独立福利）。
3. **套餐组合**：依赖 *pairing* Skill 激活；按「最小数量」计算组数，避免重复超额减免。
4. **配送费公式**：`distanceKm > 2.0 → 4.50`，否则 `2.50`；自提 `0`。
5. **积分兑换**：`userPoints >= 500` 方可兑换免费脆薯饼，兑换后扣减积分。
6. **税费**：固定 `8.5%`，基于「小计 − 折扣 − 组合返利」计算。

## 11. 小结与可扩展点

本系统将「可热更新的业务规则（Brain）」与「受保护的事务执行（Limbs）」解耦，具备良好的可演示性与安全性。后续可扩展方向：

- 将 Mock 规则引擎与真实 MiniMax 分支的折扣计算统一收敛到一份共享定价模块，消除前后端逻辑重复。
- 引入真实 Cloudflare Worker / 数据库持久化，替换内存态 `logs` 与随机订单号。
- 为 MCP 工具补充单元测试（库存、时段、互斥等边界场景）。
- 增加 `remove_item_from_cart`、营养查询等工具在前端 UI 的直接触发入口（目前主要由 AI 对话驱动）。
```
